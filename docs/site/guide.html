User guide

Everything the studio, CLI, and Figma plugin can do, in the order you'll meet it. Start the app with pnpm install && pnpm build && pnpm --filter @okeytokey/studio dev and open http://localhost:5173.

1. Concepts in two minutes

A token is a named design decision — colors.blue.500 = #3b82f6. Tokens live in groups (the dots in the path) inside token sets (files, e.g. global, semantic, dark). Every token has a $type (color, dimension, typography, shadow…), usually inherited from its group.

  • Alias: a value like {colors.blue.500} points at another token. Change the primitive, every alias follows.
  • Math: dimensions and numbers can be expressions — {spacing.base} * 2, ({a} + {b}) / 2. Units are checked; no px + rem mixing.
  • Color functions: darken({colors.blue.500}, 0.1), alpha(…, 0.5), mix(a, b, 0.5) — computed perceptually in OKLCH/OKLAB.
  • Theme: an ordered list of sets, each enabled, source, or off. Later sets override earlier ones, which is how a dark theme retints every alias.

Files are plain W3C DTCG JSON. okeytokey's extras (guidelines, lifecycle, decisions) live under $extensions["com.okeytokey"] — strip them and the file is still valid for any other DTCG tool.

2. First run

The onboarding wizard offers three starts: a starter architecture (primitive → semantic sets with light/dark themes — recommended for a first look), import of existing DTCG/Tokens Studio JSON, or connect GitHub. Your work persists in the browser (IndexedDB) automatically — no save button.

Studio follows your operating-system appearance on first use. Use the Dark mode switch at the bottom of the left sidebar to choose an appearance; the choice persists across projects and reloads. This changes Studio's interface only—it does not activate a token theme or alter token values, previews, exports, or synced files.

3. The token grid: themes as columns

The main view is a treegrid: your group hierarchy stays in the Name column (expand/collapse, drag to reorganize, arrow-key navigation), and each theme renders as a value column — Light and Dark side by side, like Figma Variables. Because DTCG stores themes as sparse override sets rather than per-mode values, the grid is honest about where each value comes from:

  • Dimmed cells are inherited — the theme resolves the base value; nothing theme-specific exists. Full-strength cells are overrides, defined in the theme's own set (hover the cell to see which set, and what it resolves to).
  • Color cells edit in place — click the hex/RGB text and type any CSS syntax, just like a string or number (Enter commits, Escape cancels). Click the swatch to open a Figma-style popover instead: native picker, free-text, OKLCH sliders, the hex / rgb / oklch notation switch, and ⤳ Reference another token… to link the cell to an existing token. Edits apply live while it's open; Escape or clicking outside closes it.
  • Editing a base cell edits the token in place. Editing an inherited cell in another theme creates a sparse override in that theme's set — only that theme changes, and your files stay clean. If the theme's override set was deleted, it's recreated automatically in the same step. Overridden cells grow a hover ↺ that removes the override and inherits again. Every edit is one undo.
  • Double-click a name to rename inline — the rename-refactor retargets every alias, math expression, and color function that pointed at it.
  • ＋ in the header adds a mode or theme: New mode creates a set for its overrides plus a theme column layered on the base, joining the same theme group; New theme only adds a theme over the existing sets. ＋ New token (footer) adds a row.
  • Keyboard: ↑/↓ move across tokens, ←/→ move the column focus, Enter edits the focused cell.
  • Composite values (typography, shadows) show a summary chip in the cell — select the row and use the inspector's dedicated composers. With no themes defined, the grid shows a single Value column.

3b. Creating and editing tokens

New token (toolbar) asks for a path like colors.brand.500 — intermediate groups are created for you, and the type follows the group you're targeting. The dialog helps with the value too, computed from what's already in your system:

  • Colors get a native picker next to the text field, plus a suggestion chip when the value is computable: a numeric path between existing scale anchors offers the interpolated step ("fits the colors.gray scale at 400"). Typing a value that already exists as a token offers to reference it instead of duplicating it.
  • Dimensions and durations suggest the next step in the group's progression (geometric or arithmetic) and fill its largest gap.
  • Font families get a dropdown of fonts already in your tokens, system stacks, and a curated Google Fonts list; font weights a named 100–900 dropdown. Free text always stays available.
  • ⤳ Reference an existing token… creates an alias directly — pick the target and the new token adopts its type. This is the fast path for semantic tokens.

Every row also carries a ⋮ menu (revealed on hover, right next to the name). On a group: New token… and New subgroup… open the dialog with the group's path locked in as a read-only prefix — you only type the new leaf — plus Sort A→Z (orders just that group's members, leaving the rest of the set as-is) and Rename group…. On a token: Rename token… and Duplicate token (a -copy sibling carrying the value, type, and description). Renaming a group cascades to every child and retargets every reference that pointed into it, and a single ⌘Z reverses the whole refactor.

Select any token to edit it in the right-hand inspector:

  • Colors: click the swatch to open your OS color picker, type any CSS syntax (hex, oklch(…), rgb(…)), or drag the OKLCH sliders. The native picker speaks opaque 6-digit hex only — use the text field or the A slider for alpha, and oklch(…) for wide-gamut values. An amber dot warns when a color exceeds the sRGB gamut. The hex / rgb / oklch switcher converts the notation (same color, different syntax) — and offers to apply the format to every other color literal in the group in one undoable step. Aliases and color functions keep their form.
  • Dimensions get the same treatment for units: a px / rem switcher converts the value at the CSS convention of 1rem = 16px (stated in the bar), then offers to convert the rest of the group. Unlike color notation this changes the number, not just the syntax — and converting a group together keeps math expressions consistent, since px and rem never mix inside an expression. Aliases and math keep their form; expressions over a converted base resolve in the new unit automatically.
  • Typography / shadows get dedicated composers; shadows support multi-layer stacks.
  • “⤳ Reference another token…” turns the value into an alias. The picker searches by resolved value too — paste #3b82f6 to find the token that holds it; click anywhere outside (or Esc) to dismiss it. While a value is an alias, “Change…” retargets the reference and “Detach” bakes the resolved value back in — detach first if you want to pick a direct color.
  • The → line under the editor always shows the fully resolved value under the active theme.

4. Organizing: rename, drag, filter

  • Rename… (inspector) renames a token or moves it to a new path, previewing every alias, math expression, and composite that will be rewritten before you confirm. This is the rename-refactor — nothing ever dangles.
  • Drag a token (or a whole group) onto another group to move it there; drop on the empty list background to move it to the root. Same refactor engine — references follow, and one ⌘Z reverses the whole move.
  • Filter (toolbar) matches names and values.
  • Sort A→Z (a set's ⋮ menu) orders every group's members by name — scale steps read 50, 100, 500 naturally, words alphabetically. Values are untouched, and one ⌘Z restores the original order. The same menu holds Rename…, Export JSON, and Delete set… (confirmed, undoable).
  • Every content edit is undoable: ⌘Z / ⇧⌘Z, 100 steps.

4b. Generating scales (color, spacing, size)

Open Generate scale steps… from the command palette (⌘K) and point it at either shape:

  • A single color token (colors.red) — the everyday case: you made one color and want its scale. The token becomes a seed step (red.500 by default, configurable) — every alias and expression referencing it follows automatically — and a full ramp from near-white to near-dark is synthesized around it, keeping its hue. With a token selected, the dialog prefills this for you.
  • A group with numeric anchors (blue.100/500/900) — the missing steps between anchors are interpolated; anchors are marked in the preview and never touched. The optional lightest/darkest end fields set the range yourself and unlock steps like 50/950 beyond your outermost anchors.

Review the preview, then apply. Everything is OKLCH-interpolated, fully deterministic (same inputs, same results, no AI), one undo removes the whole batch — including the seed rename — and generated tokens carry lineage metadata recording the generator and its anchor inputs.

For dimensions and durations there's a sibling: Generate spacing / size scale… (also ⌘K). Give it a base value (16px) and a ratio — pick a preset (minor third, major third, perfect fourth, golden, ×2) or type your own — and it builds a modular scale: value(step) = base × ratio^offset around the base step. Existing steps are kept, only the missing ones are generated, and the unit comes from your base value. Same determinism, same one-undo, same lineage.

5. Themes and theme groups

Themes live in the grid header — each is a value column. Add one with the header ＋ menu, then open the column's ⋮ menu → Edit sets… and set each set's status: Enabled (resolves and exports), Source (resolves but isn't exported — right for primitive sets), Off. Click a theme's name to make it the active theme — the inspector resolves against it — and click again to deactivate. The same ⋮ menu deletes a theme (your tokens are untouched).

Give themes a Group (e.g. mode for light/dark, brand for brand variants) and the header ＋ menu's ⊞ Generate combinations creates every combination — light / brand-b and friends — as ordinary themes you can tweak or delete. Regenerating never overwrites your edits.

6. Decision context and deprecation

The inspector's Decision context section stores why a token exists: usage guidelines, a lifecycle (draft → active → deprecated → archived), and a “replaced by” pointer. Set a replacement, hit Deprecate, and every token still aliasing the old one gets a diagnostic with a one-click “Point at … instead” fix. Deprecated tokens render struck-through in the list.

7. Diagnostics

The bar at the bottom of the list live-lints the whole document. Click a row to jump to the offending token; fixes run through the undo stack.

Rule Default Catches
no-broken-references error aliases pointing at nothing
no-reference-cycles error a → b → a, with the exact path
deprecated-usage warn aliases to deprecated tokens (auto-fixable)
no-orphan-tokens warn tokens nothing references
contrast warn WCAG 2.1 + APCA for declared fg/bg pairs (config)
naming-convention off segment patterns, per-type overrides (config)

8. Exporting code

Export CSS/SCSS/TS… (sidebar) previews and downloads: CSS custom properties (optionally keeping aliases as var() chains), SCSS map, TypeScript consts with literal types, or a Tailwind v4 @theme block — each resolved under any theme. Convert px to rem rewrites every px length in the output to rem at 1rem = 16px (shadows and composite values included; 0px collapses to 0) — format-agnostic, so it applies to whichever target you pick. For CI, the standalone CLI reads okeytokey.config.json: 

{
  "sets": ["tokens/global.json", "tokens/dark.json"],
  "themes": [ … same shape as the studio's themes … ],
  "build": {
    "outDir": "dist/tokens",
    "targets": [
      { "format": "css", "file": "tokens.css", "theme": "light", "darkTheme": "dark" },
      { "format": "ts", "file": "tokens.ts", "theme": "light" }
    ]
  }
}
pnpm okeytokey build okeytokey.config.json

The darkTheme option emits a prefers-color-scheme: dark block containing only the variables that differ.

9. GitHub sync

Sync with GitHub… takes owner, repo, branch, a path (directory of .json set files), and a fine-grained personal access token with Contents read/write on that repository.

  • Run connection doctor first: it checks token → repo → branch → path in order and tells you exactly which step failed and how to fix it.
  • Dry-run push shows the semantic diff (added/removed/renamed/changed tokens, plus how many downstream tokens change after resolution) — the push button only appears after you've seen it. Pushes land as one atomic commit.
  • Pull merges token-by-token: edits to different tokens never conflict, even in the same file. Real conflicts are picked per token — keep mine / take remote.

9b. AI provider (optional, off by default)

AI provider… (sidebar or ⌘K) configures where AI-assisted workflows run. The default is None: nothing is ever sent anywhere. Inference runs on your machine or through your own account — the dialog states which, per choice.

  • Local / OpenAI-compatible talks to any /v1/chat/completions server. Presets, ordered by setup ease: OpenRouter (your own key, no installs), LM Studio (free app; enable CORS in its server settings), and Ollama (free app; the dialog shows the exact OLLAMA_ORIGINS launch command to copy). With a local endpoint, token data never leaves your machine.
  • Anthropic uses your own sk-ant-… API key directly from the browser (Anthropic supports this; most other clouds don't, which is why OpenAI-direct isn't offered).
  • Test connection tells you exactly what was checked and what came back — wrong key, unreachable server, or missing model, with the fix.
  • Settings and keys live in this browser's localStorage (the same tradeoff as the GitHub token) and are never written into token files or Git sync.

9c. Generating semantic tokens with AI

Generate semantic tokens (AI)… (⌘K) turns primitives into a reviewed proposal. The flow is fixed: the model proposes, the schema parses, core validates, you see a diff, and only what you approve is applied.

  • Scope is a path like colors or colors.blue (comma-separate several). The dialog states exactly how many tokens will be sent — the selected subtree plus one level of referenced tokens, never the whole document.
  • Review lists every proposed operation with a checkbox. Operations that fail core validation are struck through with the reason — they can't be applied. Uncheck anything you don't want; the change count and downstream impact update live.
  • Apply lands the accepted operations as one step: a single ⌘Z reverses the whole acceptance. Malformed model output is rejected before anything runs, with the raw text available for inspection.

10. Figma plugin

Build with pnpm --filter @okeytokey/figma-plugin build, then in Figma desktop: Plugins → Development → Import plugin from manifest… and pick apps/figma-plugin/manifest.json.

  • Load DTCG JSON (export sets from the studio), select canvas nodes, pick a token and a target — fill, stroke, corner radius, padding, gap, or typography — and apply.
  • Export variables creates a variable collection with one mode per theme name (comma-separate names in the theme field). Import variables goes the other way, producing DTCG sets with a report of anything unmappable.
  • Pin theme stores the active mode in the file; new component instances automatically get it re-applied, so they never snap back to the default mode.
